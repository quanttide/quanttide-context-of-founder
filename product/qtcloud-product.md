# 量潮产品云工作语境

## 提示词

## 分析任务
对比 [文档A] 与 [文档B]，判断是否存在本质差距。

## 分析前必须执行的自我检查

### 1. 打破形态偏见
- 产品形态包括：GUI、CLI、库、API、方法论、规范、数据集。
- 目标产出是否属于上述任一？如果属于，继续。

### 2. 区分抽象层级
将每个陈述标记为：L1价值 / L2功能 / L3实现。
只有L1/L2不一致才算差距。L3实现方式不同不算。

### 3. 扩展“写作”定义
写作包括：文学、技术文档、PRD、邮件、笔记、代码注释等。
判断场景是否在定义内。

### 4. 区分本质 vs 程度
- 本质差距：核心逻辑/价值主张对立。
- 程度问题：核心一致，细节未完善。

### 5. 强制反向验证
先尝试构建“两份文档一致”的解释，至少找3条支持证据。
如果无法构建，才得出结论。

## 输出格式
- 支持一致的证据：
- 存在的程度问题（非本质）：
- 本质差距（如有）：

### 解释 

这个提示词在产品研发流程中的核心功能是：作为“认知对齐工具”或“评审前置过滤器”，服务于需求、设计、代码等多个环节的文档一致性验证。

具体来说，它的功能可以拆解为：

1. 需求评审阶段：PRD 与原型/代码的快速对账

· 当产品经理写完 PRD，开发团队给出技术方案或原型代码时，用这个提示词驱动 LLM 进行对比分析。
· 功能：快速识别哪些差异是“本质逻辑冲突”（需要开会讨论），哪些只是“实现细节未完善”（可以后续迭代），避免评审时陷入无意义的细节争论。

2. 技术设计阶段：架构文档与实现代码的一致性检查

· 架构师写出设计文档，开发人员写出核心模块的伪代码或 demo。
· 功能：用提示词让 LLM 检查代码是否体现了设计文档的价值目标和功能能力，而不是逐字匹配类名、函数名。能自动发现“设计上要求 A，代码却实现 B”的本质偏差。

3. 代码审查（Code Review）辅助

· 当 PR 描述与代码改动之间可能存在理解偏差时。
· 功能：帮助 reviewer 和 author 区分“本质逻辑错误”与“风格/实现程度问题”，减少因认知定势（如 reviewer 默认找茬）导致的无谓返工。

4. 多文档一致性维护（如 PRD → 测试用例 → 用户手册）

· 产品演进中，多个文档容易产生漂移。
· 功能：定期用该提示词让 LLM 扫描文档对，标记出需要人工介入的“本质冲突”，保持整个研发流程的信息对齐。

5. 团队认知盲区检测

· 在新人 onboarding 或需求宣讲后，用该提示词让 LLM 模拟一个“打破思维定式”的评审者。
· 功能：主动发现团队集体默认但可能错误的假设（例如“产品必须有 GUI”“PRD 里的功能名必须等于代码中的模块名”），提前纠正方向性错误。

6. 自动化 CI 流程中的轻量级验证

· 将提示词封装成脚本，在代码提交或文档更新时自动触发 LLM 检查。
· 功能：作为一道低成本的门禁，过滤掉那些明显因为抽象层级错配、元应用被忽略等问题导致的无效修改，提高人工评审的效率。
