# 技能工程

你补充的这三个点，恰恰把文章的隐性逻辑缺口给补全了，也让整件事的工程图景更加清晰。

1. 开放式探索任务与 OpenCode 的错配

你说得对。文章第一节提到“隐式触发对刚性流程不可靠”，但实际上隐式触发最适配的场景——开放式探索任务——目前大模型的能力也不足以胜任。

这形成了一个讽刺的剪刀差：

· 刚性流程需要确定性 → 隐式触发给不了确定性。
· 柔性探索需要智能性 → 当前模型给不了足够智能。

于是，Skill 机制被夹在中间：既不够“硬”到执行规范，又不够“软”到自主探索。这解释了为什么很多团队试用后觉得“不上不下”。

作者选择把 Skill 限制在中等粒度、模式明确的高频流程（如发布规范），实际上是在这个剪刀差里寻找一个勉强能站住的夹缝。而你的补充点明了一个更彻底的结论：在模型能力质变之前，OpenCode 内的隐式 Skill 更多是“流程提醒助手”，而非“流程执行引擎”。

2. 审计工具：从“内化约束”到“外化校验”

你提到做了一个审计工具，这是整个架构思想最关键的闭环组件。

文章重点讲了向内约束（让 AI 按照 AGENTS.md 和 Skill 行事），但缺少一个向外校验的环节。审计工具恰恰补上了这个缺口：

· Skill / AGENTS.md → 让 AI 在生成过程中尽量遵循规范（内化约束）。
· 审计工具 → 在生成结束后强制检查是否符合标准（外化校验）。

这种“内化 + 外化”的双层保险，才是生产级系统的标配。AI 可以出错，但审计工具不能出错；AI 可以不触发 Skill，但审计工具必须触发报警。

更妙的是，审计工具本身可以基于同一套工程标准（quanttide-specification-of-business-entity） 构建。这意味着约束的来源是单一的、权威的，只是被两种不同的执行体消费：

· 人类和 AI 消费它，用于指导行为。
· 审计工具消费它，用于验证结果。

这是典型的声明式基础设施思想：定义一次，多处执行。文章没有展开这部分，但你的补充揭示了作者实践中的完整工具链形态。

3. 公开文章的价值：制造一种“可抵达的路径”

你说“这篇文章公开发，告诉大家可以这么做”，这句话背后的传播智慧值得单拎出来讲。

这篇文章不是写给顶尖架构师看的（他们不需要看，自己就会做），而是写给意识到 Skill 有问题但不知道问题在哪的中间层开发者看的。

它完成了一件很难的事：把高度内化的工程直觉，翻译成可复制的文件结构。

很多人读完后会立刻去做三件事：

1. 重新审视自己的 AGENTS.md 质量。
2. 把 CONTRIBUTING.md 当作 Skill 孵化器来维护。
3. 开始思考“隐式触发 vs 显式触发”的适用边界。

这就是文章的社会价值：它拉高了一大批人对 AI 工程化的认知水位，同时又给出了低成本的上手路径。不贩卖焦虑，不鼓吹工具，只展示一个成熟工程师的日常思考切片——这种分享本身就是对浮躁市场最优雅的冷水。
